分析口径：
1. 你的目标不是泛泛解释“这个工序可能和什么有关”，而是给出后续系统可执行、可提问、可裁剪的规则条件。
2. 先判断该工序在当前样本范围内是稳定出现，还是只在部分样本中条件出现。
3. 先判断该工序在整体路线中的作用和所处阶段，再分析触发因素；不要预设任何固定零件类型、固定工艺链、固定因素集合或固定加工逻辑。
4. 优先判断当前差异是否已经能落成“工序在什么条件下存在”的实际工艺边界；只有样本证据支持时，才继续收敛为触发因素。
5. 不要预设所有工序都默认受材料、尺度、结构、热处理等同一组因素影响；只分析与当前工序差异真正相关的条件类别。
6. 若当前样本还不足以落成材料、结构、尺寸或要求规则，不要硬写成伪条件；应保留空 `factors` 或 `WEAK` 说明，并在 `evidence` 中明确指出缺口。
7. 即使存在名称、写法或记录差异，也不要把这类表象直接当成最终规则结论；最终结论仍应优先回到可执行的存在条件。
8. 只能根据当前任务样本和参考资料中真实出现的信息判断；不要因为工序名称、行业经验或常见套路，臆造当前任务中未出现的因素。
9. 只有在当前样本范围内没有观察到可识别分支边界时，才允许使用 `always=true`。
10. 只要存在例外样本，就不要直接输出 `always=true`；应优先分析例外边界、样本范围差异或数据问题。
11. 对可能重复出现的工序，必须结合 `sequence`、前后工序和所处阶段分别分析，不能将不同阶段的同名工序合并成一个条件。
12. 如果当前任务已经提供了可识别的因素字段，应优先使用这些字段；不要臆造只适用于某一类零件的默认字段。
13. 不要把样本中的共现关系直接当成因果关系；应优先寻找与该工序作用更直接、且对“出现 / 不出现”更有区分力的触发维度。
14. 如果能落成具体条件，就不要停留在“结构相关”“材料相关”“热处理相关”这类大类名。
15. 如果某工序只在部分样本中出现，但你暂时不能精确锁定条件，请将其标记为 `WEAK`，并把因素尽量写成便于系统继续追问用户的问题形式。
16. 如果当前样本和参考资料都不足以支持某个条件，请输出空 `factors` 或仅保留明确的 `WEAK` 说明；不要因为工序名称去猜默认因素。
17. 不要把“孔类工序默认对应孔结构”“热处理工序默认对应材料/硬度”“磨削默认对应粗糙度”这类行业常识直接写成规则，除非当前样本里真的有边界证据支持。

通用示例原则：
1. 已知工序名时，不要先重复说明加工对象存在；应优先分析“为什么这里需要这道工序”。
2. 先从工序名称和工序内容中识别“加工对象”和“加工方法”。若已能判断对象是外圆、内孔、端面、槽、螺纹、平面、型孔，方法是车、钻、镗、铰、磨、研、珩、热处理、检验或表面处理，则后续分析应直接围绕该对象和方法的边界展开。
3. 若当前差异属于分支判断，不要只停留在“材料相关”“结构相关”“要求相关”这类大类；应继续收敛到用户可确认的具体依据。
4. 若当前是特征成形类工序，应优先说明它在形成什么特征、角色、位置或边界，不要泛写“结构相关”。
5. 若当前是要求驱动型工序，应优先分析为什么需要更高等级的方法或更后段的工艺，重点看精度、形位、粗糙度、装配功能、热后精整这类直接边界。
6. 同一加工对象若使用不同方法，应区分“成形”和“精整”两种分析方式。例如车槽/铣槽优先看槽特征，磨槽则应优先判断是否由尺寸、公差、粗糙度或热后恢复要求驱动。
7. 若统一名称已经确认，不要再把“多种叫法、需结合上下文判断”作为主因写进最终结论；命名归并只能作为弱辅助信息。
8. 当工序名称已经足以说明家族，例如“终热处理”“表面处理”“内孔研磨”“车槽”，不要再把它当成通用未知工序去分析“结构/尺寸/要求”哪一类更像原因；应直接进入该家族下的专用边界分析。
