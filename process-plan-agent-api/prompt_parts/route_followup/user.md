目标：基于当前工序的弱规则审核上下文，只把当前已有问题改写得更自然、更专业。

输入包含：
1. 当前工序信息
2. 当前审核状态
3. 当前问题列表
4. 每道问题的层级、依赖关系与候选项
5. 当前已知边界、例外样本或待确认线索（若有）

请输出 JSON 对象，结构如下：
```json
{
  "questions": [
    {
      "operation_id": 1,
      "question_key": "reason_category",
      "question_text": "面向用户展示的问题文本"
    }
  ]
}
```

硬性约束：
1. 只允许改写 `question_text`，不得改变题目数量、题目 key、题目层级、候选项、候选项顺序或分支关系。
2. 不要输出 `decision`，不要判断是否停止追问；是否继续追问由规则模板和系统状态决定。
3. 不要输出 `recommended_option_values`；候选项排序由规则模板决定。
4. 只改写当前问题，不要把多个维度混问在一起。
5. 若当前是例外类问题，要保留“例外在什么条件下存在”的语义。
6. 先判断当前更像特征成形类工序还是要求驱动类工序，再决定问法。
7. 特征成形类工序优先改写成特征、角色或边界问题；要求驱动类工序优先改写成要求边界问题。
8. 名称混用、记录口径或包含关系不要抢到主路径前面。
9. 尺寸类问题尽量改写成具体口径。
10. 只围绕当前顶层主工序发问，不扩展出新工序。

少量样例（仅作辅助）：

1. 特征成形类工序先问特征角色
- 不推荐：`这道钻孔主要是因为什么精度要求才会安排？`
- 推荐：`这道钻孔存在的条件，主要依赖哪一种孔特征或孔边界？`

输入开始：
question_payload_json:
{{question_payload_json}}
