## 1. 阶段 3 目标（完成标准）

全部满足才算阶段 3 完成：

1. 第 4 步主路径改为：**定稿卡片 → 后端 `/compile` → `/validate`（或编译内嵌校验）→ 存草稿 → 模拟 → 发布**。
2. ZIP 内容与库内物化快照一致（优先后端快照，避免前后端各拼一份）。
3. 第 5 步按 V2 `input_schema.fields` 动态渲染表单；V1 仍读 `required_inputs` / `optional_inputs`。
4. 存在**项目级开关**（或等价机制）将正式生成切到 `plan_route`；默认策略可配置（建议：旧项目 V1，新项目可开 V2）。
5. 草稿不影响第 5 步；仅 `published` 包参与正式生成。
6. 工序改名不影响规则（规则只引用 `process_id`）。
7. 新增材料/特征在 `allow_custom` 或 options 配置下可不改生成代码即可录入并命中规则。
8. 生成结果可追溯：`rule_package_id` / `version` / `content_hash` / 命中轨迹摘要。

**非目标（仍属阶段 4）：** 系统/零件族模板库、跨项目模板合并。

## 5. 后端生成失败策略（阶段 3 必须写死）

1. **校验失败的草稿**：不能 publish（已有 lifecycle）。
2. **已发布但运行时校验失败**：返回 422 + validation 详情，**不要**静默退回 V1（避免「看起来成功但规则错了」）。
3. **V2 规划空工序**：422 + 诊断（缺主线规则 / 条件全未命中）；发布前包内测试应尽量阻断。
4. **项目开 V2 但只有 V1 包**：走 V1，summary 标明。
5. **影子模式（未开 V2）**：正式 V1；V2 差异写入日志或响应 `shadow_diff` 字段（可选，便于观察）。

### 6.4 端到端（阶段 3 出门条件）

对应实施方案 §12.4，阶段 3 至少跑通 1–4；第 5 条（全新材料）在动态表单 + 配置规则就绪后必过：

1. 旧项目仅 V1 → 生成不变。  
2. 新项目 V2 草稿 → 模拟通过 → 发布 → 开开关 → 生成。  
3. 改规则 → 新草稿 → 发布 → 验证新路线 → 回滚。  
4. 改工序显示名、保留 `process_id` → 结果仍含该工序。  
5. 添加代码中从未出现的材料 + 规则命中。

### 6.5 自动化最低门禁

合并阶段 3 前 CI（或本地）必须：

```bash
# 后端
cd process-plan-agent-api
pytest tests/ -q

# 前端
cd process-plan-agent-ui
npm test
npm run build
```

新增用例不得跳过；V1 基线测试失败则阻断发布。

## 8. 建议工期与风险

| 切片 | 粗估 | 主要风险 |
| --- | --- | --- |
| A | 0.5–1 天 | 契约与前端 any 漂移 |
| B | 2–4 天 | 卡片信息不够编出可执行 V2（compiler 过薄）；需补 process_id 稳定生成策略 |
