## 1. 一句话结论

ProcessMind 是一个**典型工艺规程智能体**的最小可运行交付包：从多份历史工艺规程中提炼母路线与影响因素规则，再对新产品参数做**可追溯、偏确定性**的工艺路线生成。

它已经具备可演示、可本地交接的五步业务闭环，并正在从「前端拼装 + 工序名称匹配」的规则包 V1，迁移到「可校验、可模拟、可发布」的规则包 V2 运行时。后端 V2 骨架领先于前端与正式生产切换。

## 2. 项目定位与交付形态

### 2.1 它是什么

| 维度 | 说明 |
| --- | --- |
| 产品名 | ProcessMind / 典型工艺规程智能体 |
| 业务目标 | 样本规程 → 母路线 → 规则包 → 新零件参数下的路线实例 |
| 形态 | 前端 SPA + 后端 API + SQLite 数据快照 + 启动脚本 + Docker |
| 目标用户 | 工艺工程师（主操作）、规则维护者、继续开发的工程师 |
| 部署场景 | 内网试用、研发交接、演示；非公网多租户产品 |

### 2.2 包内保留与不保留

**保留：**

1. 前端源码 `process-plan-agent-ui`
2. 后端源码 `process-plan-agent-api`
3. 运行数据快照 `data/`
4. 配置模板 `docs/配置模板/`
5. Docker 与本地启动脚本

**通常不作为干净源码交付的内容：** `node_modules`、前端 `dist`、Python 虚拟环境、历史评测/临时输出等（当前仓库里可能已有本地运行产物，如 `.runtime/`、`dist/`）。

### 2.3 技术栈一览

| 层 | 技术 |
| --- | --- |
| 前端 | Vue 3、Vue Router、TypeScript、Vite、Element Plus、Axios、Vitest |
| 后端 | FastAPI、Uvicorn、SQLAlchemy 2.x 异步、aiosqlite、Pydantic v2、httpx |
| 文档解析 | PyMuPDF、python-docx、openpyxl |

粗量级参考（分析时统计）：后端 `app/` 约 1.8 万行 Python，前端 `src/` 约 2.7 万行 Vue/TS。业务重心在规则提炼、归并、问答审核与规则包执行，而不是薄 CRUD。

## 4. 功能能力总览

### 4.1 产品主路径：五步工作流

顶栏步骤条（`App.vue`）固定为：

| 步骤 | 路由 | 页面职责 |
| --- | --- | --- |
| 1 上传 | `/upload` | 创建任务、上传典型规程与参考资料 |
| 2 路线归并 | `/extract` | LLM 提炼路线全集 + 人机归并 + 保存规范化路线 |
| 3 规则分析 | `/analysis` | 按工序段做因素/问题树审核，对照证据 |
| 4 规则定稿 | `/finalize` | 编规则包、导出 ZIP、生命周期（草稿/发布/回滚） |
| 5 路线生成 | `/generate` | 填零件参数，按已发布规则包生成工序工步树 |

### 4.2 各步能力细节

#### 第 1 步：上传任务资料

- 新建「工艺规程规则」任务（profile 当前主要为 `route_rules.document_rules`）。
- 主文件：典型工艺规程（pdf / docx / xlsx）。
- 参考资料：文字说明或上传文件（老工艺师经验、编制原则等）。
- 任务、文档、规则、生成结果按 `project_id` 隔离。
- 项目状态从 `CREATED` 到 `UPLOADED`。

#### 第 2 步：路线归并

- 异步提炼任务：`POST /api/extract/start`，状态轮询 `GET /api/extract/task-status`。
- **必须配置可用 LLM**；无 Key 时第二步直接失败。
- 从规程中抽取文档级工序明细与路线全集。
- 构建超集路线、归并建议、规范化超集路线。
- 人工审核归并建议后保存规范化路线版本（`NormalizedRouteVersion`）。
- 相关面板：源路线、归并队列、规范化路线、进度与状态卡片。

#### 第 3 步：规则分析

- 基于**已保存**的规范化路线版本工作。
- 左侧工序段列表，右侧证据与问题树。
- 因素确认/排除、段级规则审核落库。
- 配置驱动强：
  - 前端：`src/config/questionProfiles/*`、`paramQuestionStrategy`、`precisionQuestionTreeConfigs` 等
  - 后端：knowledge 文档、问题树模板路径、`param_question_*` 服务簇
- 支持证据摘录、样本对比、文档预览缓存等。

#### 第 4 步：规则定稿

- 将审核后的工序卡片编译为规则包载荷：
  - `input_schema.json`
  - `factor_dictionary.json`（可嵌入 schema/catalog）
  - `route_catalog.json`
  - `route_rules.json`
  - `rule_report.md`
- 前端构建与校验：`utils/finalizeRulePackage.ts`
- 保存：`POST /api/extract/finalized-rule-packages`
- 下载 ZIP：`utils/exportArchive.ts`（浏览器侧打 zip）
- V2 生命周期面板：草稿 / 已发布 / 历史版本、发布与回滚入口

#### 第 5 步：路线生成

- 按规则包 `input_schema` 渲染输入字段并提交。
- 设计目标：**第 5 步不由大模型实时决定工序**，由确定性规则解释/规划产出。
- 生产路径现状见第 6 节（V1 正式、V2 影子）。

### 4.3 辅助能力

- 系统设置：LLM URL / Key / Model、连接测试、模型列表
- 文档预览、PDF 分页图片
- 规则包 V2 预览 API：compile / validate / simulate（可不写库）
- 规则包持久化：按 id 查询、发布、回滚、按 id 模拟
- Docker 部署与健康检查

### 4.4 后端 API 地图（按路由前缀）

| 前缀 | 主要用途 |
| --- | --- |
| `/api/projects` | 任务列表/创建/删除、profile 列表 |
| `/api/documents` | 上传、列表、删除、预览、文件下载、PDF 页、参考资料 |
| `/api/extract` | 提炼任务、工序与明细、超集/归并/规范化路线、规则包保存与列表 |
| `/api/extract/finalized-rule-packages` | V2 compile/validate/simulate/publish/rollback 及按 id 操作 |
| `/api/generate` | factor-schema、生成路线（及历史 param 问答相关逻辑） |
| `/api/settings` | 读写设置、测连通、列模型 |
| `/` | 服务元信息与 `/docs` Swagger |

### 4.5 数据模型要点

核心表/实体围绕「项目工作空间」展开：

- `Project`：任务与状态
- `Document` / `Reference`：主文件与参考资料
- `DocumentOperationDetail`：文档级工序明细
- `Operation` / `Factor`：提炼出的工序与因素
- `RouteMergeSnapshot`：归并工作区快照
- `NormalizedRouteVersion`：规范化路线版本
- `NormalizedRouteSegmentFactorReview` / `...RuleReview`：段级审核
- `FinalizedRulePackage`：规则包版本与生命周期字段
- `GeneratedRoute`：生成结果
- `ParamRuleSnapshot` / `ParamAuditAnswer`：参数规则与审核答案（历史/并行能力）

`FinalizedRulePackage` 已具备 V2 所需字段：`schema_version`、`status`、`manifest_json`、`content_hash`、`published_by/at`、`supersedes_id`、`test_cases_json` 等。

### 4.6 知识与提示词资产

`knowledge/route_rules/` 示例：

1. 通用工艺路线基础
2. 回转类规则因素与按需工序
3. 业务场景专用知识
4. 特征分类与提问词表
5. 问题生成与循环控制规则
6. 规则输出格式规范
7. 高精度配合回转件基础知识切片（JSON）

另有 `knowledge/route_merge/` 归并规则、`prompt_parts/` 分片提示词。

### 4.7 Profile 与领域范围

`profile_registry` 当前实质注册的是：

- key：`route_rules.document_rules`
- mode：`route_rules`
- 定位：上传典型工艺规程 → 提炼母路线/工序/因素 → 生成固定工艺路线

领域知识与配置明显偏向**回转类 / 高精度配合**场景；箱体、钣金等零件族尚未作为一等 profile 扩展。

## 5. 使用方式

### 5.1 环境要求

- 本地开发：Python 3.11+、Node.js 20+、npm 10+
- Docker：Docker Engine + Compose
- 网络：第 2 步需要能访问所配置的 LLM 服务

### 5.2 本地启动

**Windows（推荐双击）：**

1. `start-windows.cmd` 启动（首次自动 bootstrap）
2. `stop-windows.cmd` 停止

**macOS：**

1. `start-macos.command` / `stop-macos.command`
2. 首次若被系统拦截，在 Finder 中右键「打开」

**终端：**

```bash
./bootstrap.sh
./start-api.sh    # http://127.0.0.1:8000  文档 /docs
./start-ui.sh     # http://127.0.0.1:5173
```

说明：

- 日志默认在 `.runtime/logs/`
- 脚本会检查 8000 / 5173 端口占用，被占用时停止启动并提示进程，不结束无关程序
- 前端开发态默认请求 `http://当前主机:8000`，可用 `VITE_API_BASE_URL` 覆盖
- 后端默认读包内 `data/`，可用 `PROCESSMIND_DATA_DIR` 等环境变量改路径

### 5.3 Docker 启动

```bash
cp .env.compose.example .env
# 编辑 LLM_API_URL / LLM_API_KEY / LLM_MODEL 等
docker compose up -d --build
```

默认 Web：`http://127.0.0.1:8080`（可用 `PROCESSMIND_WEB_PORT` 修改）。  
API 数据卷：`./data` → 容器 `/runtime-data`。

### 5.4 推荐操作员路径（首次 10 分钟级）

1. 打开系统设置，填写并测试 LLM 连通（**必做**）。
2. 第 1 步：新建任务 → 上传多份同类典型规程（建议多份，归并更稳）→ 可选参考资料。
3. 第 2 步：启动提炼 → 等待完成 → 审核归并建议 → 保存规范化路线。
4. 第 3 步：按工序段过问题树，结合证据确认/排除因素，保存审核。
5. 第 4 步：导出/保存规则包。
   - V1：保存后通常自动发布
   - V2：默认草稿，需在生命周期面板显式发布
6. 第 5 步：按表单填零件参数 → 生成路线 → 检查是否已加载规则包。

### 5.5 路径与配置约定

| 变量/路径 | 作用 |
| --- | --- |
| `PROCESSMIND_DATA_DIR` | 数据根（默认项目 `data/`） |
| `PROCESSMIND_DB_PATH` | 数据库文件 |
| `PROCESSMIND_UPLOAD_DIR` | 上传目录 |
| `PROCESSMIND_SETTINGS_PATH` | 设置文件 |
| `LLM_API_URL` / `LLM_API_KEY` / `LLM_MODEL` | 大模型（环境变量或设置页） |
| `docs/配置模板/第三步规则分析-问题树节点配置.template.json` | 第三步问题树模板 |

### 5.6 开发者调试入口

- Swagger：`http://127.0.0.1:8000/docs`
- 前端 dev：`cd process-plan-agent-ui && npm run dev`
- 后端：在 api 目录 venv 中 uvicorn 启动（与 `start-api` 脚本一致）
- 测试：
  - 后端：`pytest`（`process-plan-agent-api/tests/`）
  - 前端：`npm test`（vitest）

## 6. 架构与关键行为边界

### 6.1 端到端数据流（当前实现）

```text
上传规程
  → 文档入库 + 文件落盘
  → LLM 提炼路线全集 + 文档工序明细
  → 归并工作区 / 规范化路线版本
  → 段级因素与规则审核
  → 前端/后端构建规则包
  → finalized_rule_packages（版本递增）
  → 第 5 步读取已发布包 + 参数
  → 确定性生成工序工步（目标）
```

### 6.2 确定性 vs LLM

| 环节 | 主要机制 |
| --- | --- |
| 第 2 步提炼 | **LLM 必需** |
| 第 3 步问答 | 规则树 + 配置 + 部分 LLM 跟进 |
| 第 4 步校验/模拟 | 确定性校验器与规划器（V2） |
| 第 5 步生成 | **已发布 V2 正式走 `plan_route`**；V1 包仍走 V1 解释器 |

### 6.3 规则包 V1 vs V2（现状）

| 维度 | V1 | V2 |
| --- | --- | --- |
| 契约 | 五文件 JSON/MD，偏前端拼装 | 严格 Pydantic 契约、`process_id`、表达式 AST |
| 条件语义 | 名称匹配、字符串条件混用 | 统一 ExpressionEngine |
| 保存行为 | 保存后自动 publish | 默认 `draft`，需 publish |
| 校验 | 非空等基础校验 | 结构/引用/冲突/依赖环/包内测试 |
| 第 5 步 | **正式生产路径**（`finalized_route_generator`） | **影子验证**（`plan_route` 跑通也不替换正式结果） |
| 前端 | 导出主路径 | 生命周期面板已有；完整编译发布与动态表单未闭环 |

文档 `docs/规则包V2-通用化实施方案.md`（2026-07-14）状态摘要：

**已完成：** V1 基线测试、V2 契约与表达式引擎、校验与确定性规划、compile/validate/simulate、草稿/发布/回滚、内容哈希、每项目唯一已发布版本、生命周期面板与影子路径。

**尚未实施：** 第 4 步前端编译发布完整流程、第 5 步 V2 动态表单、正式生成路径切换、跨项目模板。

### 6.4 第 5 步生产路径（实现要点）

`POST /api/generate/` 逻辑概要：

1. 加载项目资源与已发布规则包（`load_published_rule_package`）。
2. 若 `schema_version == "2.0"`：做 V2 校验与 `plan_route` **影子验证**，但**沿用兼容/V1 路线**作为正式结果。
3. 若 V1：调用 `generate_steps_from_finalized_rule_package` 作为正式结果。
4. 若无包结果：回退到 operations 选择或最小 fallback 步骤。
5. 结果写入 `GeneratedRoute`。

因此：界面上看到「V2 影子验证通过」**不等于**生产已切到 V2 规划器。

### 6.5 服务拆分趋势

后端已从巨型路由中拆出大量服务：

- 提炼：`extraction_pipeline` / `extraction_tasks` / `route_rules_*`
- 归并：`route_merge/*`
- 问答：`param_question_*`、`param_answer_service` 等
- 规则包：`rule_packages/{compiler,validator,expression_engine,planner,lifecycle,loader,v1_adapter}`
- 生成：`finalized_route_generator`、`generate_route_result_builder` 等

`generate.py` 仍偏大，是继续下沉的重点。

## 8. 风险与限制

1. **领域偏窄**  
   单一主 profile；材料/特征/字段大量前端硬编码，跨零件族扩展成本高。

2. **V1/V2 双轨易误解**  
   生命周期 UI 与影子验证可能让人以为「已上 V2」，但正式生成仍可能走兼容路径。

3. **第 2 步强依赖 LLM**  
   无 Key / 不可达则不可用；缺少离线或纯规则兜底。

4. **安全与交接**  
   - CORS 全开、无鉴权，仅适合内网  
   - `process_settings.json` 可能含真实 API Key，分发前应脱敏  
   - SQLite 单机，无多租户隔离

5. **步骤门禁偏软**  
   顶栏主要按是否有 `project_id` 控制，不严格校验「已归并 / 已定稿 / 已发布」。

6. **可维护性**  
   `generate.py` 与部分前端 composable 职责密集；改 V2 正式路径时回归面大。

7. **运维**  
   提炼任务状态偏内存态（进程重启后依赖项目状态回推），长任务可观测性仍可加强。

## 9. 改进与演进建议

### 9.1 短期（1–2 周，高 ROI）

| 优先级 | 项 | 说明 |
| --- | --- | --- |
| P0 | 密钥脱敏 | 移出仓库、设置页掩码回显、README 标明首次必配 LLM |
| P0 | V1/V2 状态透明 | 第 5 步明确展示「正式引擎 / 影子结果」 |
| P1 | 步骤前置校验 | 无规范化路线不可进分析；无 published 包不可生成，并给跳转 |
| P1 | 黄金路径 smoke | 样例数据 + fixture/录制 LLM → 归并 → 导出 → 生成 |
| P2 | 操作员手册 | 五步说明 + 失败排查（Key、端口、PDF 解析差） |

### 9.2 中期（收口规则包 V2 通用化）

与 `规则包V2-通用化实施方案.md` 对齐：

1. 第 4 步前端走 **compile → validate → simulate → publish**，减少「只拼五文件就当生产」。
2. 第 5 步 **正式切换** `plan_route`；V1 仅 adapter 兼容历史包。
3. **input_schema 驱动动态表单**，去掉材料/特征硬编码。
4. 规则统一 **`process_id`**，消灭中文名模糊匹配副作用。
5. 生成结果返回 **package_id / version / content_hash / 命中轨迹**。
6. 继续把 `generate.py` 下沉到服务层，缩小路由文件。

### 9.3 中长期（产品化）

1. 零件族 / 企业模板：编译时物化到项目级快照（运行时仍只读一份已发布包）。
2. 多 profile：箱体、钣金等扩展 `profile_registry`，避免回转件假设全局化。
3. 提炼可观测性：进度、token、失败重试、按文档重提。
4. 内网权限与审计：项目级隔离、`created_by` / `published_by` 强化。
5. 评测集：固定规程样本 + 期望工序序列，回归 planner 与表达式引擎。

### 9.4 建议演进顺序

```text
安全脱敏与使用说明
  → 步骤门禁 + V2 状态透明
  → 前端编译/发布闭环
  → 第 5 步正式切 V2
  → 动态表单去掉硬编码
  → 模板与多零件族
```

**不建议**并行大改第 1–3 步文档理解管线。应先稳定「定稿 → 生成」契约，再扩展前半段智能化。

## 10. 对接手者的实用建议

### 10.1 第一天

1. 读 `README.md` 与本文档第 5 节，本地跑通启动。
2. 配置 LLM 并在设置页测连通。
3. 用 `data/` 里已有项目或上传 1–2 份 PDF 走完五步，建立体感。
4. 打开 `/docs` 对照前端 `src/api/*`。

### 10.2 改规则行为时

1. 先判断改的是 **知识/提示词**、**问题树配置**，还是 **规则包执行语义**。
2. 动 V2 契约时以 `app/services/rule_packages/contracts.py` 与测试为准。
3. 动 V1 生成时以 `finalized_route_generator` + `test_finalized_route_generator_v1` 为准。
4. 改前端导出时同步看 `finalizeRulePackage.ts` 与后端 save 校验。

### 10.3 改 UI 流程时

1. 步骤跳转统一带 `project_id` query（`useCurrentProject`）。
2. 归并从分析返回时注意 `resume=route_merge` 等 query 约定。
3. 大块业务状态优先落在 composable，而不是继续堆 view。

## 11. 总结

ProcessMind 已经是一个**可演示、可交接的垂直领域工作台**：五步业务闭环完整，数据模型围绕「样本规程 → 母路线 → 规则包 → 实例生成」，知识与配置资产齐全，并具备 Docker/本地双启动路径。

**当前最大价值**在第 2–4 步的人机协同提炼与规则沉淀。  
**当前最大技术债与战略机会**都在第 4–5 步的规则包通用化：后端 V2 骨架超前，前端编译发布与生产切换尚未收口。

若继续开发，最划算的投入是：

1. 安全与状态透明（立刻降低误用与泄漏风险）  
2. 收口 V2 发布与正式生成  
3. 用 input_schema 去掉配置硬编码  

而不是再堆平行页面或过早重写前三步。
